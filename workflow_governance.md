# Workflow Governance

Status:
ACTIVE

Purpose:

Define the governance doctrine for DesignOS architecture, repository ownership, workflow evolution, and retrieval behavior.

This repository governs:

* Architecture Evolution
* Repository Ownership
* Dependency Management
* Retrieval Governance
* Source Budget Management

This repository does not govern:

* Art Direction
* Design Interpretation
* Workflow Execution
* Prompt Assets
* Template Assets

---

# Core Philosophy

## Consumer AI First

DesignOS is designed primarily for AI consumption.

Priority Order:

1. Retrieval Accuracy
2. Retrieval Consistency
3. Retrieval Efficiency
4. Human Convenience

AI retrieval quality takes precedence.

---

## Character Truth First

Character Truth remains authoritative.

DesignOS consumes Character Truth.

DesignOS does not create Character Truth.

---

# Architecture Doctrine

## Architecture Fast Track

Question
↓
Evidence
↓
Audit
↓
Recommendation
↓
Decision
↓
Implementation

Evidence precedes implementation.

---

## Current State First

All architecture decisions begin with:

Current State
↓
Gap
↓
Proposal

Proposal may not precede current-state analysis.

---

# Repository Doctrine

## Single Responsibility

Each repository owns one primary responsibility.

Ownership overlap is prohibited.

---

## Ownership Separation

Ownership exists once.

Dependency does not create ownership.

Consumption does not create ownership.

Interpretation does not create ownership.

---

## Repository Creation

Preferred Order:

Expand Existing Repository
↓
Patch Existing Repository
↓
Merge Existing Repository
↓
Create New Repository

Repository creation requires justification.

---

# Retrieval Governance

## Retrieval Efficiency First

Consumers should retrieve only information required for the active task.

Selective retrieval is preferred.

---

## Source Budget

Source budget is finite.

Repository proliferation reduces retrieval efficiency.

New repositories require retrieval justification.

---

## Runtime vs Knowledge

Runtime repositories should contain:

* Ownership Maps
* Retrieval Maps
* Registry Information

Knowledge repositories should contain:

* Doctrine
* Frameworks
* Specifications
* Detailed Guidance
